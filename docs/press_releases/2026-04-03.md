# Lit Protocol Adds Third-Party Module Imports to Lit Actions: Thousands of npm Packages Now Available Inside Verifiable Compute

**April 3, 2026 — San Francisco, CA**

Lit Protocol today announced **module imports for Lit Actions**, a major expansion of its programmable compute platform. Developers can now import third-party ES modules from npm directly inside Lit Actions, with every dependency verified by SHA-384 integrity hashes before execution. The feature is available immediately on Lit Chipotle.

## Why This Matters

Lit Actions are immutable JavaScript programs that run inside Trusted Execution Environments and can sign data with network-managed wallets. Until today, developers were limited to the libraries bundled with the runtime — ethers.js and the Lit Actions SDK. Anything else had to be copy-pasted or pre-bundled into the action code.

That constraint is removed. A single import statement now gives actions access to thousands of production npm packages: validation libraries like Zod and AJV, date utilities like date-fns, encoding tools like cbor-x, and more. This turns Lit Actions from a signing-focused runtime into a general-purpose serverless compute platform with cryptographic superpowers.

```javascript
import { z } from "zod@3.22.4/+esm";

async function main({ pkpId, userData }) {
  const schema = z.object({
    email: z.string().email(),
    amount: z.number().positive(),
  });

  const result = schema.safeParse(userData);
  if (!result.success) {
    return { error: "Validation failed", issues: result.error.issues };
  }

  const wallet = new ethers.Wallet(
    await Lit.Actions.getPrivateKey({ pkpId })
  );
  const signature = await wallet.signMessage(JSON.stringify(result.data));

  return { validated: result.data, signature };
}
```

## How It Works

Developers write a simple npm-style specifier with a pinned version (e.g. `zod@3.22.4/+esm`), and the runtime automatically resolves it to [jsDelivr](https://www.jsdelivr.com/), a public CDN that serves npm packages as ready-to-run ES modules. The runtime enforces integrity at every step:

- **URL allowlist** — Only `https://cdn.jsdelivr.net/` origins are accepted. All other sources are rejected before any network request is made.
- **SHA-384 integrity verification** — Every module is hashed and compared against an integrity manifest. Developers can also declare the expected hash directly in the import statement (`import { z } from "zod@3.22.4/+esm#sha384-..."`) for self-contained verification with no external lockfile. If the content has been tampered with, the action does not execute.
- **Trust-on-first-use with up to four-way verification** — When a module is imported for the first time, the runtime fetches it twice from the CDN independently, computes the SHA-384 of each response, verifies both against jsDelivr's SRI hash header, and if the import includes an inline `#sha384-` hash, verifies against that as well. The module is accepted only if all checks agree, and the verified hash is pinned for all future requests.
- **No redirects** — HTTP redirects are blocked entirely, preventing the CDN from being used as a proxy to load code from unauthorized sources.
- **Sandboxing** — Imported code runs in the same hardened Deno/V8 sandbox as the rest of the action. No filesystem access, no subprocess execution, no native code.

## Why jsDelivr

The choice of CDN was driven by the security requirements of running third-party code inside a cryptographic signing environment:

1. **Immutability** — The same URL must serve the exact same bytes forever. jsDelivr with pinned versions guarantees this. Once a version is published to npm, the content at `https://cdn.jsdelivr.net/npm/zod@3.22.4/+esm` never changes. Chipotle enforces this guarantee by validating the SHA-384 hash of every module on every fetch, so even if a CDN were to serve altered content, the integrity check would catch it and reject the module before execution.

2. **No server-side transformation** — jsDelivr serves the original files from the npm tarball. It does not perform bundling, transpilation, or CJS-to-ESM conversion. What the package author published is exactly what gets served. Alternatives like esm.sh and Skypack perform server-side transformations that cannot be audited or pinned.

3. **Version pinning** — Exact version URLs (`@3.22.4`) make the resolved content fully deterministic, which is required for integrity hashing and reproducible action behavior.

4. **Infrastructure** — jsDelivr is backed by a multi-CDN infrastructure (Cloudflare, Fastly, and others) with built-in SRI hash support and high global availability.

## What Developers Can Build

Module imports unlock new categories of Lit Actions that were previously impractical:

- **Input validation** — Use Zod, AJV, or Superstruct to validate data before signing it, ensuring only well-formed data receives a cryptographic attestation.
- **Data encoding** — Encode payloads as CBOR, protobuf, or other binary formats before signing for efficient on-chain storage.
- **Date and time proofs** — Generate timezone-aware timestamps with date-fns and sign them to produce verifiable time attestations.
- **Complex business logic** — Build multi-step workflows using utility libraries, with signed outputs at each stage.

Any npm package that ships ES modules and does not depend on Node.js built-ins or native addons is compatible. This includes most modern JavaScript libraries.

## Availability

Module imports are available now on Lit Chipotle. No migration or configuration change is required for existing actions. To start using imports, add an `import` statement with a pinned jsDelivr URL to your action code.

Full documentation, including import syntax, compatibility requirements, and working examples, is available in the [Module Imports guide](https://developer.litprotocol.com/lit-actions/imports).
